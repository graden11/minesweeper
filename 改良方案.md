# 改良方案

> 2026-06-01 全项目代码审查，按严重程度分级。

---

## 🔴 严重问题

### 1. Router 动态路由路径参数丢失

- **文件**：`HttpServer/src/http/Router.cpp:70`
- **问题**：路径参数提取到 `newReq`，但回调传的是原始 `req`，参数全部丢失，动态路由（如 `/:id`）核心功能失效
- **修改**：`callback(req, resp)` → `callback(newReq, resp)`

### 2. MemorySessionStorage 无锁

- **文件**：`HttpServer/include/session/SessionStorage.h:27`
- **问题**：`unordered_map` 在 muduo 多 IO 线程模型下无任何 mutex 保护，每次请求都会 `save()` 直接写入 map，data race（UB）
- **修改**：加 `std::shared_mutex`

### 3. RedisSessionStorage 跨线程共享 redisContext

- **文件**：`HttpServer/include/session/RedisSessionStorage.h:22`
- **问题**：hiredis 的 `redisContext*` 本身不是线程安全的，多线程下 `save/load/remove` 并发调用同一 `ctx_` 会导致数据损坏
- **修改**：改为每线程一个 context（thread_local）或使用连接池加锁

### 4. sslConns_ map 跨线程无同步

- **文件**：`HttpServer/include/http/HttpServer.h:154`
- **问题**：`onConnection` 在主循环线程 insert，`onMessage` 在 worker 线程 find。`std::map` 并发 insert + find 是未定义行为
- **修改**：加 mutex，或利用 muduo 保证每连接回调在同一 event loop 的前提做更精细控制

### 5. LoginHandler onlineUsers_ / loginSessions_ 竞态窗口

- **文件**：`WebApps/InferenceServer/src/handlers/LoginHandler.cpp:40-58`
- **问题**：`onlineUsers_` 检查和 `loginSessions_` 更新在两个独立锁区间，并发登录同一用户会产生孤儿 session（前一个被覆盖后无人引用）
- **修改**：合并到一个临界区，使用 `scoped_lock` 同时锁两个 mutex

### 6. RequestBatcher.stop() 丢弃队列导致 broken_promise

- **文件**：`WebApps/InferenceServer/src/RequestBatcher.cpp:25-31`
- **问题**：`stop()` 仅置 `running_=false` 并 join worker，队列中剩余的 `promise` 被析构，`future.get()` 抛 `std::future_error(broken_promise)`，可能触发进程崩溃
- **修改**：stop 时排空队列，为每项 set_value 返回错误 JSON，再 join

### 7. ONNX 引擎多线程并发 Run() 无锁

- **文件**：`WebApps/InferenceServer/src/ResNet50Engine.cpp:71-76`
- **问题**：多线程同时调用同一 `Ort::Session::Run()`，TRT 版有 `gpu_mutex_` 但 ONNX 版没有
- **修改**：加 `std::mutex`（类似 TRT 版），或为每线程创建独立 `Ort::Session`

### 8. 数据库密码硬编码

- **文件**：`WebApps/InferenceServer/config.json:20-21`、`HttpServer/include/utils/ConfigLoader.h:23-24`、`docker-compose.yml:6`、`docker-compose.cpu.yml:6`
- **问题**：`root/root` 明文写在 Git 跟踪文件中，任何拿到仓库的人都能连接数据库
- **修改**：从 config.json 和 ConfigLoader.h 移除默认密码；docker-compose 通过 `.env` 文件注入 `MYSQL_ROOT_PASSWORD`，`.env` 加入 `.gitignore`

### 9. 本地 + Docker 双重编译

- **文件**：`start.sh:108-114`、`build.sh:18-25`、`Dockerfile`、`Dockerfile.cpu`
- **问题**：脚本本地 cmake/make，Dockerfile 多阶段构建又 cmake/make 一次，本地产物从未被使用，浪费编译时间
- **修改**：二选一 —— (a) 本地编译 + Dockerfile COPY 二进制；(b) 全部交给 Docker 构建。建议在开发环境选 (a) 加快迭代

### 10. CLAUDE.md 文档与 Dockerfile 矛盾

- **文件**：`CLAUDE.md:156`
- **问题**：文档说 "Dockerfile needs pre-built binary, must run cmake + make before docker compose build"，但 Dockerfile 是多阶段构建，完全不需要预编译
- **修改**：删除该行，改为描述多阶段构建的实际行为

## 🟡 中等问题

### 11. friend class 滥用，强耦合

- **文件**：`WebApps/InferenceServer/include/InferenceServer.h:109-119`
- **问题**：14 个 handler 全部声明 friend，LoginHandler/LogoutHandler 直接操作内部状态和 mutex
- **修改**：抽 `SessionService` 类封装在线用户管理、踢人下线、登录态维护，暴露线程安全的公共 API

### 12. base64Decode / readFile 代码重复

- **文件**：`PredictHandler.cpp:14-50`、`BatchPredictHandler.cpp:13-49`
- **问题**：匿名命名空间中约 70 行完全重复
- **修改**：抽出到独立 `ImageUtil.h/cpp` 公共模块

### 13. ModelUnloadHandler 使用 param1/param2 而非命名参数

- **文件**：`ModelUnloadHandler.cpp:27-28`
- **问题**：用 `req.getPathParameters("param1")` / `("param2")` 而非 `("name")` / `("version")`，路由位序变化即失效
- **修改**：路由注册使用命名参数 `:name/:version` 并以此取值

### 14. LogoutHandler stoi 未登录时抛异常 → 500

- **文件**：`LogoutHandler.cpp:22`
- **问题**：`std::stoi(session->getValue("userId"))` 当 session 无 userId 时抛 `std::invalid_argument`
- **修改**：先检查 session 中是否存在 userId 且 isLoggedIn=="true"

### 15. RegisterHandler JSON parse 无 try/catch

- **文件**：`RegisterHandler.cpp:7-9`
- **问题**：`json::parse(req.getBody())` 无异常处理，非法 JSON 直接抛异常导致 500
- **修改**：加 try/catch，返回有意义的 4xx 错误

### 16. ReadyHandler 每次请求新建 Redis 连接

- **文件**：`ReadyHandler.cpp:39-45`
- **问题**：每次 /ready 都 `redisConnectWithTimeout` + `redisFree`，无连接池
- **修改**：复用已有 RedisSessionStorage 连接，或缓存检查结果设置短 TTL

### 17. 生产日志打印完整 HTTP 响应体

- **文件**：`HttpServer/src/http/HttpServer.cpp:200`
- **问题**：`LOG_INFO << "Sending response:\n" << buf.toStringPiece().as_string()` 将所有响应（含敏感数据）写入日志
- **修改**：移除或改为 TRACE 级别

### 18. SSL 初始化失败直接 abort()

- **文件**：`HttpServer/src/http/HttpServer.cpp:89`
- **问题**：生产环境 SSL 配置错误不应崩溃整个进程
- **修改**：抛出异常或返回错误让上层决定

### 19. DbConnectionPool 归还连接时捕获裸 this

- **文件**：`HttpServer/src/utils/db/DbConnectionPool.cpp:85-89`
- **问题**：deleter 捕获 `this` 裸指针，连接池单例销毁后已外借的连接归还时访问已析构的队列（UAF）
- **修改**：使用 `weak_ptr<DbConnectionPool>` 或确保生命周期严格有序

### 20. MetricsMiddleware 延迟计算偏高

- **文件**：`HttpServer/src/middleware/MetricsMiddleware.cpp:22`
- **问题**：用 `request.receiveTime()` 起算，包含 HTTP 解析和 muduo 调度耗时
- **修改**：改为 `muduo::Timestamp::now()`

### 21. SessionManager 每次请求无条件持久化

- **文件**：`HttpServer/src/session/SessionManager.cpp:41`
- **问题**：即使 session 未变更，每次请求也 `storage_->save()`，Redis 后端产生无谓网络 IO
- **修改**：Session 加 dirty flag，仅脏数据才写回

### 22. ConfigLoader 实现全放头文件

- **文件**：`HttpServer/include/utils/ConfigLoader.h`
- **问题**：近 200 行配置解析逻辑 inline 在 .h 中，编译扇出大
- **修改**：拆分为声明（.h）+ 实现（.cpp）

### 23. CMakeLists 硬编码 x86_64 系统路径

- **文件**：`CMakeLists.txt:20-21, 129, 138-139`
- **问题**：MySQL 头文件、libfmt.so、TensorRT/CUDA 库路径写死 `/usr/lib/x86_64-linux-gnu/...`，ARM 或非 Debian 发行版无法编译
- **修改**：使用 `find_package(fmt REQUIRED)` 和 `find_library`

### 24. httpserver 服务缺少健康检查

- **文件**：`docker-compose.yml`、`docker-compose.cpu.yml`
- **问题**：仅 MySQL/Redis 有 healthcheck，`depends_on: service_healthy` 无法验证应用层就绪
- **修改**：添加 healthcheck 调用 `/health` 端点

### 25. Dockerfile 用 sed 硬改 config.json

- **文件**：`Dockerfile:58`、`Dockerfile.cpu:48`
- **问题**：sed 替换路径，格式变化则静默失败
- **修改**：改用环境变量 `MODELS_PATH` 在启动时指定路径

## 🟢 建议优化

- **模型目录三处不一致**：README (`WebApps/InferenceServer/models/`) vs start.sh (`models/`) vs Dockerfile.cpu (`COPY models/`) —— 统一为 `WebApps/InferenceServer/models/`
- **现代 C++ 实践**：返回值加 `[[nodiscard]]`；枚举改 `enum class`；参数用 `string_view` 替代 const string&；deleter 用 `weak_ptr` 替代裸 this capture
- **动态路由 O(n) 线性扫描**：`regexHandlers_` 和 `regexCallbacks_` 每次请求遍历全部正则做匹配，可按 method 分组或使用 trie
- **getUserCount() 每次查库**（`InferenceServer.h:96-106`）：高频 `getBackendData()` 触发 `SELECT COUNT(*)`，加 30 秒缓存
- **MenuHandler 字符串注入**（`MenuHandler.cpp:42-48`）：sql_quote userId 虽是 int 暂无可利用性，但模式有 XSS 风险，建议前端通过 API 获取用户信息
- **Middleware::setNext() 死代码**（`middleware/Middleware.h:23`）：链式设计未被使用，MiddlewareChain 走 vector 迭代，移除冗余设计
- **ModelLoadHandler 与 InferenceServer::initialize() 引擎构造重复**：`ModelLoadHandler.cpp:100-135` 与 `InferenceServer.cpp:95-116` 的 loadEngine lambda 几乎一致，抽 `ModelFactory::loadModel()` 统一入口
- **stop.bat / start.bat 与 "Linux only" 矛盾**：README 声明 Linux only，但项目有 Windows 批处理文件，需澄清 WSL 桥接关系
