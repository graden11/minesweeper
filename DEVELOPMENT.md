# Development Direction

## 核心定位：推理服务框架 vs 推理引擎

**推理引擎 (Inference Engine)** 解决的是"如何让**一个模型**在**一块硬件**上跑得最快"——图优化、算子融合、量化、Kernel 调优。

**推理服务框架 (Inference Serving Framework)** 解决的是"如何让**多个模型**在**高并发网络环境**下，作为一个稳定、高性能的**服务**运行"——网络 IO 管理、请求调度、内存管理、数据流水线、模型版本管理、监控。

TensorRT 是火箭发动机，本项目是把发动机、燃料系统、导航、减震、乘客座椅整合在一起的整辆赛车。不是你造了一个比 TensorRT 更快的推理引擎，而是你做了一个 TensorRT 们不做的、面向特定场景的完整系统。

---

## 创新空间：TensorRT 们不做的"苦活"

TensorRT 的定位是通用、普适，留下了大量针对特定场景的优化空间：

### 1. 网络层与推理层的"鸿沟"

- TensorRT 管的事：你给我一个 Tensor，我帮你算。
- 你做的事：从网络收到一堆字节流，经过解析、解码、预处理，变出一个 Tensor，再喂给 TensorRT。
- 内存池和零拷贝，就是在优化"变出 Tensor"这个过程——让网络层的数据零拷贝直接变成推理引擎的输入。

### 2. 消费级硬件的"资源约束"

- TensorRT 的默认优化倾向于 A100/H100 等专业卡，显存大、带宽高。
- 本项目专为消费级 GPU（如 RTX 5060）设计。在显存有限、带宽紧张的情况下，内存池可以做到更精细的显存管理，避免 OOM 和碎片。
- 这是"资源受限"约束下的创新。

### 3. 系统层面的"全链路优化"

- 单个引擎只优化推理本身。
- 你优化的是整个端到端链路：网络 IO → 解码 → 预处理 → 推理 → 后处理 → 返回结果。
- 内存池和异步流水线在系统层面做全局优化，单个推理引擎做不到。

## 论文叙事定位

> 我设计并实现了一个面向消费级 GPU 的轻量级推理服务框架。该框架通过自研内存池和全链路零拷贝设计，解决了通用推理服务框架（如 Triton）在消费级硬件上因频繁内存分配和数据拷贝导致的性能瓶颈。实验表明，在 RTX 5060 等消费级 GPU 上，该框架在处理高并发推理请求时，相比 Triton/FastAPI 方案，延迟降低 X%，吞吐量提升 Y%。

核心贡献不在引擎层（不是"我写了一个比 TensorRT 更快的推理引擎"），而在系统层——网络 IO、内存管理、调度策略。

## 差异化路线

### 竞争对手不是 TensorRT，而是：

- **Triton Inference Server** — 功能全但重，设计目标面向数据中心
- **TorchServe** — PyTorch 生态绑定
- **FastAPI + ONNX** — 没有推理优化的通用 Web 框架方案
- **Ray Serve** — 分布式优先，轻量场景开销大

### 差异化优势：

- **轻量** — 单二进制，无外部依赖链
- **紧耦合** — 网络层与推理层共享内存，不做拆包再装包
- **消费级硬件极致优化** — 针对小显存、低带宽做内存策略
- **C++ 全链路** — 无 Python/GC 开销

## 进阶方向

### 1. 为内存池实现 TensorRT Plugin

TensorRT 允许自定义插件。将内存池封装成 TensorRT Plugin，直接管理推理过程中的中间张量内存。从"框架层"切入"引擎层"，技术深度大幅提升。

### 2. 针对 Blackwell 架构 FP4 的特定 Kernel 优化

RTX 5060 支持 FP4，是较新的特性，TensorRT 对 FP4 的支持可能尚未达到最优。针对特定模型（如 LLaMA）手写 CUDA Kernel，若能在推理环节比 TensorRT 自动调优快 10%，就是顶会级别的创新。
